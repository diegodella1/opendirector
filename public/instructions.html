What is OpenDirector

OpenDirector is an open-source live TV production system designed as a replacement for CuEZ. It enables producers to build and manage show rundowns from any browser, while operators execute the show in real-time against vMix.

The system comprises two components that work together:

Component Purpose Runs On
Webapp Browser Rundown editor, teleprompter, Go Live production panel, media management Any modern browser (desktop, tablet, mobile)
Automator Windows Connects to the server, receives commands, and executes them against vMix via TCP The same Windows PC running vMix
ⓘ
The server (Next.js on a Raspberry Pi or any host) acts as the central hub. All clients connect to it. vMix is never exposed to the internet — only the Automator talks to vMix, always over the local network.

System Architecture

From anywhere (internet or LAN): [Producer] ──HTTPS/WSS──> [Server] ──WebSocket──> [Automator] ──TCP──> [vMix] browser Next.js Tauri desktop app 127.0.0.1:8099 [Talent] ──HTTPS/WSS──> [Server] prompter (browser) [Go Live] ──WSS──> [Server] <──WSS── [Automator] producer monitor bidirectional state sync

Communication Channels

The WebSocket connection carries five prioritized channels:

PriorityChannelDirectionPurpose
0 (highest) execution Server ↔ Automator CUE, NEXT, STOP, RESET commands and ACKs
1 signals Server → Prompter Countdown, wrap, stretch, go, custom messages
2 tally Automator → All PGM/PVW state, overlays, recording, streaming
3 rundown Server ↔ Automator Content sync (blocks, elements, edits)
4 prompter Host ↔ Displays Scroll position, speed, auto-scroll state

Higher-priority messages are never dropped or delayed. Lower-priority channels may discard stale data under back-pressure to ensure execution commands always get through.

Team Roles

Producer

The producer builds and directs the show:

  • Creates shows, blocks (segments), and elements (clips, lower thirds, graphics)
  • Configures GT Templates for vMix lower third inputs
  • Writes teleprompter scripts per block
  • Uploads media assets (video, images, audio)
  • During the show: monitors execution, edits lower third text on-the-fly, sends signals to talent

Operator

The operator runs the show against vMix:

  • Installs the Automator on the vMix PC
  • Connects to the server and selects the show
  • Navigates blocks and CUEs each element
  • Media syncs automatically to the local machine
  • Can run in Rehearsal (dry run) or Live mode

Talent

The talent reads the teleprompter:

  • Opens the prompter URL in any browser (tablet, phone, desktop)
  • Sees the current block's script with synchronized scroll
  • Receives visual signals from the producer (countdown, wrap, stretch, go)
  • Works offline if the server goes down

Show Lifecycle

Every show progresses through a defined state machine. This ensures an orderly workflow from preparation to archival.

Draft → Ready → Rehearsal → Live → Archived
StateDescriptionEditingAutomator
Draft Initial state. Work in progress. Full editing allowed Cannot connect
Ready Reviewed and prepared for rehearsal or broadcast. Full editing allowed Can connect
Rehearsal Dry run. Automator executes everything but does not send commands to vMix. Timing is measured. Limited editing Dry-run execution
Live On air. Real execution against vMix. Text-only edits (lower thirds) Real execution
Archived Completed show. Media remains on disk. Shows archived >90 days: export & delete suggested. Read-only Cannot connect

Producer Setup

The producer works entirely in the browser. No software installation is required.

  1. Open the webapp Navigate to the server URL in any modern browser (Chrome, Firefox, Edge, Safari).
  2. Create a show Enter a name and click New Show. Optionally, select a template to start from a pre-built rundown structure.
  3. Build the rundown Add blocks (segments), then populate each block with elements — clips, lower thirds, graphics, audio, and production notes. See Building a Rundown.
  4. Configure GT Templates Set up the mapping between your lower-third designs and vMix GT Title inputs. See GT Templates.
  5. Upload media Drag and drop video, image, and audio files into the media pool. The system validates codecs and generates thumbnails automatically.
  6. Write prompter scripts Each block has a script field. Use markup conventions for pauses, cues, and formatting. See Prompter Configuration.
  7. Set the show to Ready Once the rundown is complete, transition the show from Draft to Ready. This allows the Automator to connect.
  8. Go Live Open the Go Live panel to monitor execution in real-time, send signals, and make text edits during the show.

Operator Setup

  1. Install the Automator Download the MSI from the download page and run the installer. The app installs as a standard Windows desktop application.
  2. Prepare vMix Ensure vMix is running and the TCP API is enabled:
    vMix → Settings → Web Controller → Enable.
    Create any GT Title inputs you will use (e.g., LT_STANDARD with fields Headline.Text and Description.Text).
    Create clip pool inputs named CLIP_A and CLIP_B (or whatever names are configured in the show).
  3. Launch the Automator and connect to the server Enter the Server URL (e.g., https://your-server.com or http://192.168.1.14:3000).
    The Automator loads the list of available shows. Select the target show.
  4. Connect to vMix Enter the vMix host and port:
    • Host: 127.0.0.1 (localhost, since the Automator runs on the same PC as vMix)
    • Port: 8099 (vMix default TCP API port)
  5. Wait for media sync Media files referenced in the show are automatically downloaded to the local machine (default: C:\OpenDirector\Media).
    The Media Sync panel shows per-file progress. Wait until all items show synced.
    Files are verified with SHA256 checksums. If a download is corrupted, it re-downloads automatically.
  6. Run the pre-flight check The Automator validates that all referenced vMix inputs exist and media is loaded. Review any warnings before proceeding. See Pre-Flight Checklist.
  7. Operate the show Navigate blocks with Space (next) / Backspace (prev).
    Press Enter to CUE the selected element — the Automator sends the corresponding commands to vMix.
    Use F1–F8 for step buttons on multi-step elements.
ⓘ
One Automator per show. The server enforces this. If another Automator is already connected to the show, you will see the existing connection's IP and be blocked. Exception: if the previous Automator disconnected less than 30 seconds ago, a reconnect is allowed.

Talent / Prompter Setup

  1. Open the prompter link The producer generates a prompter URL (with embedded session token) or QR code. Open it in any browser. The prompter works on tablets, phones, PCs, and dedicated prompter displays.
  2. Go fullscreen Tap/click to enter fullscreen mode. The prompter is a PWA with landscape orientation lock and Wake Lock (screen stays on).
  3. Control scroll Multiple control methods are available:
    • Bluetooth foot pedal / remote keyboard — sends PageDown, Space, arrow keys
    • Companion device — open ?mode=control on a phone, ?mode=display on the main screen
    • Auto-scroll — configurable speed (0–100), tap to pause/resume
    • Touch drag — direct drag on touchscreen
  4. Watch for signals The producer can send visual overlays: countdown timer, WRAP (red, flashing), STRETCH (blue), STANDBY (yellow), GO (green, 3 seconds), or custom text.
✓
Mirror mode: For glass prompters, enable mirror mode. The script text is flipped horizontally, but signal overlays remain readable (they are double-mirrored).

Building a Rundown

A rundown is the complete structure of a show, organized into blocks (segments) that contain elements (individual items to execute).

Blocks

Blocks are the top-level segments of your show (e.g., "Opening", "Interview", "Weather", "Closing"). Each block contains:

PropertyDescription
NameSegment name visible in the rundown and Automator
Estimated DurationExpected length in seconds. Used for timing calculations and overrun warnings.
CamerasArray of camera names assigned to this segment (e.g., ["cam_diego", "cam_martin"])
ScriptTeleprompter text for this segment (with markup)
NotesProduction notes (not visible on air or prompter)
Statuspending → on_air → done (or skipped)

Blocks can be reordered via drag-and-drop. The position is automatically maintained.

Script Markup

The teleprompter script supports these conventions:

MarkupRenders As
[PAUSE]Visual pause indicator for the talent
[VTR: clip name]Video tape roll cue
[CUT TO CAM X]Camera cut note
[SOT]Sound on tape
[VO]Voice over
---Visual separator
**text**Bold / emphasis
(instruction)Grayed-out note (not read aloud)

GT Templates (Lower Third Configuration)

GT Templates define how your lower thirds in the webapp map to GT Title inputs in vMix. This is the bridge between the rundown editor and vMix's graphics system.

How It Works

  1. Create a GT Template in the webapp Give it a descriptive name (e.g., "Standard Name+Title", "Breaking News", "Quote").
  2. Configure the vMix mapping Set the Input Key (the key name of the GT Title input in vMix, e.g., LT_STANDARD), the overlay number (which vMix overlay channel to use, typically 2), and define the fields (e.g., Headline.Text = "Name", Description.Text = "Title").
  3. Add lower third elements to blocks In each block, add a Lower Third element and select the GT Template. Fill in the text values for each field.
  4. CUE in the Automator When the operator presses CUE, the Automator sends the following commands to vMix:
> FUNCTION SetText Input=LT_STANDARD&SelectedName=Headline.Text&Value=John Smith > FUNCTION SetText Input=LT_STANDARD&SelectedName=Description.Text&Value=Senior Producer > FUNCTION OverlayInput2In Input=LT_STANDARD

vMix Requirements for GT Templates

⚠
Critical setup in vMix:
  • The input must be a GT Title (not a standard input or Title input)
  • The input Key must exactly match what is configured in the webapp (e.g., LT_STANDARD)
  • Field names must be exact (e.g., Headline.Text, not headline or Headline)
  • To find field names: right-click the GT input in vMix → Title Editor → inspect each text block's name

Multiple Lower Third Styles

Create one GT Template per visual style. Each template points to a different GT Title input in vMix with its own design, fields, and overlay channel. For example:

Media Management

Upload (Producer → Server)

Drag and drop files into the media pool in the editor. The server processes each file automatically:

  • Storage: saved as data/shows/{show_id}/media/{uuid}.{ext}
  • Thumbnail: generated via ffmpeg (first frame for video, resized for images)
  • Metadata: duration, resolution, and codec extracted via ffprobe
  • Checksum: SHA256 computed for integrity verification during sync
  • Codec validation: ffprobe checks codec compatibility with vMix

vMix-Compatible Codecs

CategorySafe Values
Video CodecsH.264, H.265/HEVC, MPEG-2, ProRes, DNxHD
Containers.mp4, .mov, .avi, .mxf, .mpg
AudioAAC, MP3, WAV, PCM
Images.jpg, .png, .bmp, .tga
ⓘ
Incompatible codecs (e.g., VP9/WebM) generate a warning but do not block the upload. The warning reads: "webm with codec vp9 may not work in vMix. Recommended: MP4 H.264/H.265."

Download (Server → Automator)

When the Automator connects, it automatically syncs all media referenced in the rundown:

  • HTTP GET with Range headers (resumable downloads)
  • Up to 3 concurrent downloads
  • SHA256 checksum verification after each download (auto-retry on mismatch)
  • Per-file progress bar in the Automator UI
  • Post-download: registers local path in SQLite, attempts to load as vMix input (FUNCTION AddInput)

Control Modes

Foot Pedal / Remote Keyboard

A Bluetooth foot pedal or wireless keyboard sends keystrokes (PageDown, Space, arrows) to control scrolling. Ideal for camera-mounted prompters.

Companion Device

Open ?mode=control on a phone (the controller) and ?mode=display on the main screen. The phone controls scroll, the display follows. If the controller disconnects for >5 seconds, the display falls back to auto-scroll.

Auto-Scroll

Configurable speed (0–100). Tap the display to pause/resume. Server broadcasts position every 200ms; clients interpolate locally at 60fps for smooth rendering.

Touch Direct

Drag directly on the touchscreen to scroll. Best for tablet setups at a news desk.

Error Visibility

Errors from the Automator propagate to the Go Live panel via WebSocket:

  • Critical errors: Persistent red banner at the top + audible alert. Requires manual acknowledgment.
  • Warnings: Yellow badge, auto-dismisses after 10 seconds.

Execution & Cue

The execution engine in the Automator processes commands in a strict pipeline:

  1. Command received From local UI (keyboard/mouse) or remote via WebSocket.
  2. Deduplication Each command carries an idempotencyKey (UUID). Duplicate keys within 30 seconds are ignored.
  3. Validation Checks that the input exists in vMix, media is synced, and the show mode allows execution.
  4. Variable resolution Replaces template variables: {{clip_pool}} → current pool key, {{media_path}} → local file path, {{title}}/{{subtitle}} → element text.
  5. Action chain execution Runs the element's action chain in order, respecting delays between steps. A mutex ensures only one command executes at a time (5s timeout).
  6. State update + notification Updates local state, writes to the execution log, and sends ACK + state update via WebSocket.

Action Phases

stepManually triggered by operatorVisible button (PLAY, IN, OUT) with color and optional hotkey (F1–F8)

A/B Clip Pool Alternation

The Automator alternates between two clip inputs (Pool A and Pool B) for seamless playback. While one clip plays on air, the next clip is pre-loaded into the inactive pool. This enables smooth transitions without loading delays.

Offline Mode

Both the Automator and the Prompter are designed to function when the server goes offline.

Automator Offline

  • The complete rundown is cached in a local SQLite database
  • Execution continues normally against vMix using cached data
  • All actions are logged to a local event store
  • On reconnect, a reconciliation protocol merges state:
    • Content: server wins (full sync if version changed)
    • Execution state: Automator wins (the show continued offline)
    • Execution log: merged, deduplicated by idempotencyKey
  • Reconnect uses exponential backoff: 500ms → 1s → 2s → 4s → 8s → 15s max

Prompter Offline

  • All scripts are cached in IndexedDB on first load
  • Still works: auto-scroll, foot pedal, keyboard, touch, block navigation, mirror mode, wake lock, fullscreen
  • Does not work: companion device sync, signals from producer, live text edits, block sync from execution channel
  • Companion device disconnected >5s: display automatically falls back to auto-scroll
  • Discrete yellow "OFFLINE" banner in the top corner

Rehearsal Mode

Rehearsal mode is a full dry run of your show. The Automator processes every command identically to Live mode, but does not send any commands to vMix.

What Works in Rehearsal

  • CUE, NEXT, PREV, STOP, RESET — all commands are processed
  • Action chains execute (with delays) and timing is measured
  • Execution log records everything with type rehearsal
  • Tally updates are simulated
  • Signals to talent work normally
  • Prompter syncs to block changes

What Doesn't Happen

  • No TCP commands are sent to vMix
  • vMix state does not change
  • No actual video/audio playback
✓
Best practice: Always run at least one full rehearsal before going live. Use the timing data to calibrate block durations and identify issues with the rundown.

Test Procedures

Use these step-by-step procedures to validate each component of the system before a production.

Test 1: End-to-End Connectivity

Goal: Verify that all components can communicate.

  1. Open the webapp and create a test show with one block and one clip element.
  2. Set the show to Ready.
  3. Open the Automator and connect to the server. Verify the show appears in the list.
  4. Connect the Automator to vMix. Verify "vMix: Connected" in the status bar.
  5. Open the Go Live panel. Verify "Automator: Connected" in the header.
  6. Open the prompter URL. Verify "Connected" state (no offline banner).
  7. Pass: All four components show connected status.

Test 2: Media Sync

Goal: Verify media uploads and syncs to the Automator correctly.

  1. Upload a small test video (MP4 H.264, <10 seconds) to the show's media pool.
  2. Attach the video to a clip element in the rundown.
  3. In the Automator, observe the Media Sync panel. The file should start downloading.
  4. Wait for status to show "synced".
  5. Verify the file appears in the local media folder (default: C:\OpenDirector\Media).
  6. Verify the Automator attempted to load it as a vMix input.
  7. Pass: Media synced, checksum verified, input loaded in vMix.

Test 3: CUE Execution (Clip)

Goal: Verify that CUE-ing a clip element sends the correct commands to vMix.

  1. Ensure a clip element with media is in the first block.
  2. Set the show to Live (or use Rehearsal to skip vMix).
  3. In the Automator, select the clip element and press Enter (CUE).
  4. Observe the execution log — it should show the action chain (e.g., ListAdd, CutDirect, Play).
  5. If in Live mode, verify vMix shows the clip on PGM.
  6. Check Go Live panel — the execution log should mirror the Automator's log.
  7. Pass: Clip plays in vMix, execution log shows success on both ends.

Test 4: Lower Third with GT Template

Goal: Verify GT Template lower thirds set text and activate overlay in vMix.

  1. Create a GT Template mapped to a GT Title input in vMix (e.g., LT_STANDARD with Headline.Text and Description.Text).
  2. Add a Lower Third element to a block, select the GT Template, and fill in text values.
  3. In the Automator (Live mode), CUE the lower third element.
  4. Verify in vMix:
    • The GT Title input's text fields are updated with the correct values
    • The configured overlay channel activates with the input
  5. Pass: Correct text appears in the lower third on vMix output.

Test 5: Prompter Signals

Goal: Verify signals from the producer reach the prompter display.

  1. Open the Go Live panel and the prompter side by side (or on separate devices).
  2. Click 30s Countdown in the Signals section.
  3. Verify the prompter shows a yellow "30 SECONDS" overlay that counts down.
  4. Click WRAP. Verify the prompter shows a red flashing "WRAP" overlay.
  5. Click Clear. Verify the overlay disappears.
  6. Send a custom message. Verify it appears as typed.
  7. Pass: All signal types appear correctly on the prompter.

Test 6: Block Navigation & Timing

Goal: Verify block progression and timing in Rehearsal mode.

  1. Create a show with 3 blocks, each with estimated durations.
  2. Set the show to Rehearsal.
  3. In the Automator, press Space to advance through blocks.
  4. Verify Go Live shows the correct block highlighted as "on air".
  5. Verify the block timer counts up and compares against the estimated duration.
  6. Verify the prompter advances to each block's script.
  7. Complete all blocks. Verify the show progress bar reaches 100%.
  8. Pass: Smooth block progression, accurate timing, prompter syncs.

Test 7: Offline Resilience

Goal: Verify the Automator continues operating if the server goes down.

  1. Connect the Automator to a show and sync all media.
  2. Disconnect the server (stop the Next.js process or disconnect the network).
  3. In the Automator, verify it shows "Reconnecting..." status.
  4. CUE elements and navigate blocks. Verify vMix receives commands normally.
  5. Restart the server. Verify the Automator reconnects and reconciles state.
  6. Pass: Uninterrupted operation during server outage, clean reconnection.

Test 8: Panic / Emergency

Goal: Verify the PANIC button cuts to a safe input immediately.

  1. Configure overrun_safe_input_key in the show config (e.g., a logo or color bars input in vMix).
  2. During live execution, press F12 (PANIC).
  3. Verify vMix immediately cuts to the safe input.
  4. Verify the Go Live panel shows the emergency action in the execution log.
  5. Pass: Instant cut to safe content, logged in execution log.

Troubleshooting

Automator cannot connect to the server

  • Verify the server URL is correct and accessible from the vMix PC (try opening it in a browser).
  • Check firewall rules — the server port (default 3000) must be accessible.
  • If using HTTPS, verify the SSL certificate is valid.
  • Ensure the show is in Ready, Rehearsal, or Live state (cannot connect to Draft or Archived shows).

Automator cannot connect to vMix

  • Verify vMix is running on the specified host.
  • Check that the TCP API is enabled: vMix → Settings → Web Controller → Enable.
  • Default port is 8099. If changed in vMix, update the show config.
  • If connecting remotely (not localhost), check Windows Firewall allows inbound on port 8099.
  • The Automator reconnects automatically with backoff (1s, 2s, 4s, 8s max). Wait for it.

"AUTOMATOR_ALREADY_CONNECTED" error

  • Only one Automator can connect to a show at a time.
  • If the previous Automator crashed, wait 30 seconds for the server to detect the stale connection.
  • The error message includes the IP of the existing Automator.

Lower third text doesn't update in vMix

  • Verify the input is a GT Title, not a regular Title or other input type.
  • Verify the input Key in vMix matches exactly (case-sensitive).
  • Verify field names are exact: Headline.Text not headline. Check in vMix Title Editor.
  • Check the Automator execution log for SetText errors.

Media won't sync or shows error

  • Checksum mismatch: The file re-downloads automatically. If it persists, re-upload the file on the server.
  • Codec warning: The file uploaded but may not play in vMix. Re-encode to MP4 H.264.
  • Download stuck: Check network connectivity. The Automator resumes downloads with Range headers.
  • vMix won't load the input: The file may be corrupted or in an unsupported format. Test by manually adding it as an input in vMix.

Prompter is out of sync

  • Verify the prompter is connected (no yellow "OFFLINE" banner).
  • If using companion device mode, ensure the controller device is on the same network.
  • The server sends scroll position every 200ms. Clients interpolate at 60fps. Minor visual lag (<200ms) is normal.
  • Try refreshing the prompter browser tab.

Optimistic locking conflict (409 error)

  • This occurs when two editors modify the same block/element simultaneously.
  • The webapp automatically re-fetches the latest version and shows a toast: "This block was edited. Reloading..."
  • Your changes may be lost. Re-apply them after the reload.
  • In practice, this is rare since shows typically have one producer editing at a time.

vMix command errors during live show

  • Errors appear on the element with a red background and tooltip showing the error detail.
  • Use [RETRY] to re-attempt the command or [SKIP] to move on.
  • Critical errors propagate to the Go Live panel as a red banner with an audible alert.
  • Press F12 (PANIC) to immediately cut to a safe input if needed.
  • Check the execution log in both the Automator (Ctrl+L) and Go Live panel for details.

WebSocket disconnections

  • Heartbeat: ping every 3 seconds. If 2 pings go unanswered (6 seconds), the connection is considered lost.
  • Reconnect is automatic with exponential backoff: 500ms → 1s → 2s → 4s → 8s → 15s max.
  • After reconnect, the Automator runs the reconciliation protocol (content from server, execution state from Automator).
  • Rate limit: 10 messages/second per connection. Exceeding this may cause disconnection.

Known Limitations (v0.1)

LimitationDetails
Single-server architectureOne server instance, multiple clients. No clustering or HA.
One Automator per showEnforced by the server. Cannot have multiple operators on the same show.
Automator: Windows onlyvMix only runs on Windows, so the Automator targets Windows exclusively.
No automatic conflict mergeOptimistic locking detects conflicts but does not auto-merge. Last writer wins after reload.
